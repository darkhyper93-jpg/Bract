## [2026-06-07] apps/api "build" script no emite archivos compilados
**Problema:** `apps/api/package.json` define `"build": "tsc --noEmit"`, igual que `typecheck`. La flag `--noEmit` impide que TypeScript emita archivos en `dist/`, lo cual hace imposible ejecutar `node dist/server.js` en producción.
**Causa:** El script `build` fue configurado para validación de tipos, no para compilación real.
**Solución:** Cambiado a `"build": "prisma generate && tsc"` y agregado `"start": "node dist/server.js"`. `prisma generate` corre primero para que el cliente Prisma exista en `node_modules/.prisma/client/` antes de compilar. El script `typecheck` mantiene `tsc --noEmit` para validación sin output.
**Lección:** `typecheck` = validar tipos sin output. `build` = compilar con output. En imágenes Docker limpias, `prisma generate` debe correr antes de `tsc` o el import de `@prisma/client` puede fallar.

## [2026-06-07] nginx:alpine no incluye ngx_headers_more
**Problema:** La especificación usa `more_clear_headers Server;` para eliminar el header `Server` de las respuestas Nginx. Este directivo requiere el módulo `ngx_headers_more`, que no está incluido en la imagen `nginx:1.25-alpine`.
**Causa:** La imagen oficial de Nginx Alpine es minimal; módulos extras requieren compilación o imagen alternativa (nginx-extras, openresty).
**Solución:** Se eliminó la directiva `more_clear_headers`. La directiva `server_tokens off` ya presente oculta la versión de Nginx (ej: `nginx/1.25.3` → `nginx`). Si se necesita eliminar el header `Server` completamente, migrar a `openresty:alpine` o compilar Nginx con `--add-module=ngx_headers_more`.
**Lección:** Verificar disponibilidad de módulos Nginx antes de especificarlos. Las imágenes Alpine son minimal por diseño.

## [2026-06-07] BullMQ reemplazado por ejecución síncrona para MVP
**Problema:** BullMQ requiere conexión TCP persistente (ioredis). El plan Free de Upstash Redis limita a 1 conexión TCP simultánea, causando que los jobs se encolen pero nunca se procesen en producción.
**Causa:** Upstash Free usa un proxy HTTP que no soporta todos los comandos de Redis que BullMQ necesita para mantener workers activos.
**Solución:** Workers convertidos a ejecución síncrona directa: notification.producer.ts crea notificaciones directamente en Prisma; email jobs ya llamaban a Resend directamente (sin cambios); cleanup reemplazado por setInterval nativo en server.ts cada 15 minutos. Los archivos *.worker.ts y queues.ts se conservan sin modificar para reactivar BullMQ fácilmente cuando el tráfico justifique Upstash Pay-as-you-go.
**Lección:** Para MVPs, la ejecución síncrona es preferible a colas si el volumen es bajo. Las colas añaden valor a partir de ~1000 jobs/hora o cuando la latencia de respuesta es crítica.

## [2026-06-07] Arquitectura de deploy cambiada de K8s+Docker a Render
**Problema:** El README §12 especifica K8s + Nginx + Docker para producción, lo cual requiere un servidor VPS propio, nginx, certbot y mantenimiento de infraestructura.
**Causa:** El usuario necesita deploys simples vía git push sin administrar infraestructura.
**Solución:** Se adopta Render (PaaS) para el API (Web Service) y el frontend (Static Site). Render maneja TLS, health checks, rolling deploys y scaling. Los archivos Docker y nginx.conf se mantienen en el repo como fallback para deploy manual si se necesita. El deploy es: push a main → GitHub Actions CI → Render Deploy Hook → `prisma migrate deploy` automático.
**Lección:** Para startups y MVPs, PaaS (Render, Railway, Fly.io) > K8s propio. K8s tiene sentido cuando el costo del PaaS supera el costo del equipo de infraestructura.

**Problema:** Las 3 features de estudio (planner/flashcards/chat) necesitan un contrato de datos compartido (entidades + DTOs de I/O) consumido por API y web, antes de que B/C/D/E implementen.

- **Tipos de DTO de input = `z.infer<typeof schema>`** exportados desde el `.schema.ts` (sin duplicar interfaces → sin drift). Las entidades de salida sí son `interface` en `.types.ts`.
- **`Subject.color` = paleta cerrada `z.enum(SUBJECT_COLORS)`** — 8 hex curados armonizados con los tokens del design system (§9.2 dark-first): brand indigo + estados info/success/warning/error + 3 acentos (purple/pink/teal). Exportada para reuso del frontend (swatches del selector). Descartado regex hex libre por la regla §3.3 "paleta permitida".
- **Tablas-hijo sin DTO de `userId`:** `StudyPlanItem`/`ChatMessage` se scopean por su padre (§3.4); sus schemas validan solo `planId`/`sessionId` y el body del bloque/mensaje.
- **Contratos de salida de la IA NO van acá:** los define y valida el Agente B (Apéndice C). A entrega solo entidades + DTOs de request.
- **Enums espejados de Prisma como `export enum` en `.types.ts`** (patrón de `Role`/`NotificationType`): los 6 enums de producto (`TopicStatus`, `TopicDifficulty`, `StudyPlanStatus`, `StudyPlanItemStatus`, `FlashcardSource`, `ChatRole`) replican exactamente los valores del `schema.prisma`. Zod los consume con `z.nativeEnum(...)` → una sola lista de valores, sin divergencia entre DB, validación y tipos.
- **Paginación selectiva en listados:** se paginan los de crecimiento no acotado (`chat sessions`, `flashcards?topicId`, `flashcards/due` con `limit`); `subjects`/`topics` NO se paginan porque el planner los consume como árbol completo (`SubjectWithTopics`) y paginarlos rompería el render día-por-día.
**Lección:** El contrato compartido se entrega como *entidades de salida* (interfaces, fechas ISO) + *DTOs de input* (z.infer). Mantener una sola fuente de verdad por DTO (el schema Zod) y por enum (Prisma → `z.nativeEnum`) evita que tipo y validación se desincronicen. Tras tocar `packages/shared`, rebuildear su `dist` (export condicional `node`→`dist`) para no dejar el runtime desfasado para el agente siguiente.

**Causa:** Agente E del PLAN_AGENTES (Fase 12, Apéndice C). Depende de A (tipos `@bract/shared`) y B (`lib/ai`: `streamChatReply`, `assembleStudentContext`).
**Solución:** Módulo `apps/api/src/modules/chat/` (`chat.repository` → `chat.service` → `chat.controller` → `chat.routes`, montado en `/api/v1`). Decisiones tomadas:
- **DESVIACIÓN — el endpoint de mensaje NO usa el envelope `{ success, data }`:** `POST /chat/sessions/:id/messages` responde **Server-Sent Events** (`text/event-stream`). El streaming token-a-token es incompatible con un body JSON único; es la excepción documentada (ya anticipada por B, que diseñó `chatReply` como `AsyncGenerator`). Los otros 4 endpoints (listar/crear/obtener/borrar sesión) sí usan el envelope normal.
- **Formato SSE exacto** (eventos nombrados + `data` JSON; `data` JSON-encodeado para que newlines/Unicode queden seguros en una sola línea):
  ```
  event: meta\ndata: {"sessionId":"...","userMessageId":"...","title":"..."}\n\n
  event: token\ndata: {"text":"<delta>"}\n\n      (uno por cada delta del proveedor)
  event: done\ndata: {"messageId":"<assistantId>"}\n\n
  event: error\ndata: {"code":"AI_UNAVAILABLE","message":"..."}\n\n   (si falla ya streameando)
  ```
  Headers: `Content-Type: text/event-stream`, `Cache-Control: no-cache, no-transform`, `Connection: keep-alive`, `X-Accel-Buffering: no` (mata el buffering de proxy en Render/Nginx → tokens en tiempo real).
- **Separación de capas para el stream:** el **service** expone `streamMessage()` como `AsyncGenerator<ChatStreamEvent>` (HTTP-agnóstico: emite `{type:'meta'|'token'|'done'}`); el **controller** es la ÚNICA capa que toca `res` (serializa cada evento a un frame SSE). El service nunca recibe `req`/`res`.
- **Errores ANTES vs DESPUÉS del primer frame:** el generador valida ownership y `isAIConfigured()` ANTES del primer `yield` (y antes de persistir el mensaje del usuario). Como un `AsyncGenerator` no corre su cuerpo hasta el primer `.next()`, esos throws caen en el `for await` del controller con `started=false` → `next(err)` → **JSON** (`NOT_FOUND` 404 / `AI_UNAVAILABLE` 503). Una vez enviado el `meta` (headers 200), un fallo del proveedor ya no puede cambiar el status → sale como `event: error`.
- **Persistencia del reply (lo más frágil — el guardado al final):** el generador acumula los deltas en `acc` y persiste el `ChatMessage` del assistant en un bloque **`finally`**, no en el happy-path. Así un mismo bloque cubre los dos finales: (a) **stream completo** → `streamEnded=true`, el `finally` crea el mensaje con todo `acc` y toca `updatedAt`, y RECIÉN DESPUÉS se emite `done` con el `messageId` real; (b) **disconnect a mitad** → el controller llama `gen.return()` en `res.on('close')`, lo que corre el `finally` con el `acc` parcial (>0) y lo persiste igual (registrado, no a medias-perdido), sin emitir `done`. Guardas: persiste solo si `acc.length>0`; el `try` alrededor del `createMessage` logguea con Winston si la DB falla (no tira el proceso).
- **Abort real e INMEDIATO del proveedor en disconnect (AbortController atado al close):** el controller crea un `AbortController` y llama `aiAbort.abort()` en `res.on('close')`. El `signal` baja por `chatService.streamMessage(..., signal)` → `streamChatReply(input, signal)` (cambio **aditivo y retrocompatible** en B — coordinación E↔B, mismo patrón que C exponiendo `generateStudyPlanBaseline`) → `client.messages.stream(params, { signal })`, que el SDK 0.104.1 ata a su `AbortController` interno y aborta el fetch al instante. El service trata el throw por abort como corte ESPERADO (`if (!signal?.aborted) throw err`): no relanza (el controller NO emite `event: error`), y el `finally` persiste el parcial. **Por qué no alcanzaba solo `gen.return()`:** verifiqué el SDK (`lib/MessageStream.js`: el `[Symbol.asyncIterator]().return()` llama `this.abort()`), así que cortar el `for await` SÍ aborta — pero `.return()` se encola detrás del `.next()` en vuelo, por lo que el abort recién se inyecta en el **próximo límite de token** (podía generar ~1 token más). Con el `AbortController` el corte es inmediato (0 tokens extra).
- **Ownership de `ChatMessage` vía el padre (§3.4):** `ChatMessage` no tiene `userId`. `chat.repository.findOwner(sessionId)` trae `{ id, userId, title }` y el service compara contra `req.user.id` → `NOT_FOUND` si no pertenece (mismo patrón que C con `StudyPlanItem` vía `plan.userId`). Borrar/obtener sesión scopean por `{ id, userId }`.
- **Contexto del estudiante REUSANDO el árbol del planner:** el service llama `plannerService.listSubjects(userId)` (→ `SubjectWithTopics[]`, fuente única materias→temas, alineado con F) y lo pasa a `assembleStudentContext` de B. El acotado por tokens (resumen, ≤15 temas/materia, próximo examen) ya vive en B; E no lo duplica.
- **Historial acotado:** se mandan al modelo los últimos `MAX_HISTORY_MESSAGES=20` turnos user/assistant (constante, configurable), traídos newest-first y revertidos a cronológico, excluyendo el mensaje actual y los `SYSTEM`. El contexto del estudiante va aparte (no cuenta contra ese cap).
- **`title` derivado del 1er mensaje:** si `session.title===null` al primer mensaje, se setea con el `content` (espacios colapsados, truncado a 60 con elipsis) y viaja en el `meta` para que el front actualice la lista al vuelo. Editable después no está en alcance de E.
- **Sin env vars nuevas, sin modelos nuevos, sin deps nuevas:** `AI_API_KEY` la introdujo B; `ChatSession`/`ChatMessage` los aplicó A con `db push`. E NO requiere `db push`. `pnpm -r typecheck` verde (api + web).
**Lección:** Para streaming sobre Express, mantené la separación de capas haciendo que el service sea un `AsyncGenerator` de eventos de dominio y que el controller sea el único que escribe en `res`. Persistí el reply en un `finally` (no en el happy-path): un solo lugar cubre completar y desconectar, y nunca perdés lo ya generado. Chequeá disponibilidad de IA ANTES del primer frame para poder devolver un 503 limpio; después del primer byte el status ya está sellado y los errores van como `event: error`.

## [2026-06-11] Agente F — Integración y contexto compartido (efecto SRS cruzado + invalidaciones)
**Problema:** Las 3 secciones (planner/flashcards/chat) debían compartir datos y contexto EN VIVO: completar/cambiar el estado de un tema en el planner tiene que reflejarse, sin recargar a mano, en (a) el recálculo del plan, (b) la frecuencia SRS de las flashcards de ese tema, y (c) el contexto que ve el chat; las flashcards solo sobre temas del planner; una sola fuente de verdad para materias/temas/progreso.
**Causa:** Agente F del PLAN_AGENTES (Fase 12). Depende de C/D/E (ya mergeados). El usuario fijó dos requisitos: (1) coupling limpio — el planner NO toca tablas de flashcards; (2) semántica explícita y testeada del efecto de "completar tema" sobre el SRS.
**Solución:** Cero estado duplicado a nivel de cliente y un efecto de dominio cruzado en el backend. Decisiones tomadas:
- **Fuente de verdad única — ya estaba bien montada, no se duplicó:** el árbol materias→temas vive en una sola query (`queryKeys.planner.subjects`); flashcards la consume con el MISMO `useSubjects` del planner (cache compartida) y el chat la arma server-side con `plannerService.listSubjects`. F NO introdujo una segunda copia.
- **El chat NO necesita invalidación de cliente:** ensambla su contexto en el backend en cada mensaje, así que el próximo mensaje ya ve el progreso real (temas pendientes/completados, próximo examen). "El chat referencia el progreso real" queda satisfecho por construcción (decisión de E reusando `plannerService.listSubjects`), sin nada que sincronizar en el front.
- **REGLA DELIBERADA del efecto SRS (requisito 2 del usuario, Opción B):** un tema en rotación de estudio (`IN_PROGRESS`/`COMPLETED`) tiene sus flashcards **ACTIVAS** en el SRS (entran al `due`) para reforzar la retención; un tema `PENDING` las **PAUSA** (salen del `due`). El efecto **solo mueve `dueDate`**; **preserva `ease`/`intervalDays`/`reps`** (no pisa lo aprendido por repasos reales). Es idempotente. La pausa fija `dueDate` a un centinela inalcanzable (`SRS_PAUSED_DUE_DATE` = año 9999, en `srs.ts`); "pausada" se detecta con `dueDate ≥ SRS_PAUSED_THRESHOLD` (año 9000, margen amplio sobre cualquier intervalo SM-2 real). **Activar reincorpora SOLO las cartas pausadas** (centinela → `now`), dejando intacto el schedule de las que ya estaban activas (`updateMany` filtra por el umbral). DECISIÓN del centinela vs columna nueva: una columna `srsPausedAt` sería más explícita pero exige `db push` (cambio de schema) y tocar la query `due` de D; el centinela mantiene el efecto autocontenido en la app, sin migración. Trade-off documentado; si en v2 se quiere robustez total, migrar a columna.
- **Coupling limpio (requisito 1 del usuario):** el efecto vive en `flashcardService.onTopicStatusChanged(userId, topicId, status)` — el flashcard module es dueño de su data. `plannerService.updateTopicStatus` **delega** ahí (un `await`), nunca toca tablas de flashcards. Dirección de import unidireccional `planner → flashcard` (sin ciclo: `flashcard.service` no importa planner; `chat.service` ya importaba planner). El repo de flashcards expone `pauseSrsByTopic`/`activateSrsByTopic` (solo Prisma `updateMany`, scope por `userId` denormalizado §3.4).
- **Invalidaciones cruzadas centralizadas (front):** `apps/web/src/lib/invalidateStudyContext.ts` concentra el grafo en un solo lugar: `invalidateAfterTopicStatusChange(qc, topicId)` (plan + árbol + `flashcards.byTopic` + `flashcards.due`) e `invalidateAfterTreeChange(qc)` (plan + árbol + `flashcards.all()`, porque borrar materia/tema cascadea a flashcards). `useTopicMutations` y `useSubjectMutations` consumen esos helpers en vez de alcanzar keys de flashcards ad hoc.
- **Boundary explícito:** el efecto reacciona a la TRANSICIÓN de `Topic.status`; la creación de cartas (Agente D) sigue naciendo activa (`dueDate=now`), no se tocó. Pausar/activar es la reacción deliberada al cambio de estado, no un invariante recomputado en cada create.
- **Sin env vars nuevas, sin modelos nuevos, sin deps nuevas, sin `db push`** (el efecto solo escribe `dueDate` en filas existentes). `pnpm -r typecheck` verde (shared+api+web); `vitest run` 39/39 (4 tests nuevos de `onTopicStatusChanged` + 1 aserción de delegación en `planner.service.test`).
**Lección:** El "contexto compartido en vivo" se logra con UNA fuente de verdad (no copiar el árbol) + efectos de dominio cruzados detrás del service dueño de cada data (coupling limpio: el planner delega, no alcanza tablas ajenas) + un grafo de invalidaciones React Query centralizado en un módulo. Lo que se lee server-side por request (contexto del chat) no necesita invalidación de cliente. Un efecto reversible que solo mueve `dueDate` (sin tocar ease/interval/reps) implementa "frecuencia SRS" sin migración ni riesgo de pisar el aprendizaje del usuario.
